# Описание архитектуры
Проект представляет собой систему сбора и обработки данных аккселерометра на базе gRPC. Система спроектирована для работы в фоновом режиме под управлением systemd (systemd-демоны).

## Компоненты
- Узел А (Сенсор/Клиент): Эмулирует работу аппаратного датчика. Генерирует пакеты данных (AccelPacket) и отправляет их на сервер в виде потока (BidiStream). Также принимает обработанные данные обратно для записи результатов в файл.
- Узел B (Процессор/Клиент): Выступает в роли вычислительного узла. Принимает сырые данные от сервера, вычисляет модуль вектора ускорения и отправляет результат (AccelModule) обратно на сервер.
- Сервер (Центральный узел): Реализован на базе Asynchronous Callback API gRPC. Он фильтрует дубликаты и обеспечивает маршрутизацию данных между Узлом А и Узлом В.

## Безопасность
- Транспортный уровень (mTLS): Все соединения защищены протоколом TLS 1.3. Используется взаимная аутентификация: сервер и клиенты проверяют сертификаты друг друга, подписанные доверенным корневым сертификатом.
- Прикладной уровень (API-key): Дополнительная авторизация осуществляется через передачу API-ключа в метаданных gRPC. Сервер разрывает соединение, если ключ отсутствует или не совпадает с конфигурационным. Ключ хранится в конфигурационном файле и может быть изменен.

## Эмуляция сенсора
Узел A отправляет данные, полученные от эмулятора аккселерометра. Для реализации эмулятора был реализован класс `AccelEmulator`. Частота генерации данных эмулятором может быть настроена с помощью редактирования конфигурационного файла **config.yaml**.

## Логирование
Каждый из узлов логирует полученные/отправленные сообщения, подключения, ошибки и т.д. Узел A дополнительно производит запись полученных сообщений в файл **accel/module.log**.

## Версионирование
Каждое сообщение содержит поле version. В случае получения сообщения будущей версии протокола, узлы игнорируют неизвестные поля и выводят предупреждение в лог.

## Конфигурационный файл
Для настройки некоторых параметров работы программы был создан файл **config.yaml**, который содержит часттоту генерации данных сенсором (Гц), количество попыток переподключения при разрыве соединения, временной интервал между ними (мс) и API-ключ для авторизации через метаданные **gRPC**.

  
# Инструкция по сборке
Перед началом сборки необходимо установить библиотеки:

``` bash
sudo apt install \
    libprotobuf-dev protobuf-compiler \
    libgrpc++-dev protobuf-compiler-grpc \
    libyaml-cpp-dev \
    libspdlog-dev \
    libabsl-dev
```

Далее создаем папку для сборки в корне репозитория и собираем проект с помощью **CMake**:

```bash
mkdir build && cd build
cmake ..
make
```

> **Примечание:** Для возможности запуска системных демонов через **systemd** соответствующие файлы узлов **<имя_файла>.service** помещаются в системные директории при сборке **автоматически**. При удалении проекта желательно удалить эти файлы:
``` bash
trash ~/.config/systemd/user/accel_*.service
```

# Инструкция по генерации сертификатов
Сертификаты генерируются автоматически при сборке с **CMake**. Но, если необходимо их перегенерировать, можно воспользоваться скриптом **gen_certs.sh** в корне репозитория.

**API-ключ** уже указан в файле конфигурации **config.yaml**. При желании с помощью **openssl** можно сгенерировать новый, после чего указав его в **config.yaml**:

```bash
openssl rand -base64 32
```

# Примеры запуска
Запустить собранные узлы можно двумя способами: как обычные процессы или как **systemd**-демоны.

Видеоролик с примером запуска демонов находится в директории `video`.

## Запуск узлов как обычных процессов
После сборки исполняемые файлы помещаются в соответствующих поддиректориях каталога **build**:
- **build/server/server** - сервер
- **build/node_a/node_a** - узел A
- **build/node_b/node_b** - узел B

Запускать следует из корня проекта. Каждому из узлов в качестве аргумента командной строки передается номер порта. Ниже приведены примеры запуска (где каждая команда запускается в отдельном терминале):

```bash
./build/server/server 50051
./build/node_a/node_a 50051
./build/node_b/node_b 50051
```

В случае такого запуска логи узлов будут выводиться в терминал.

## Запуск узлов как systemd-демонов
Для работы **systemd**-демона необходимы файлы конфигурации сервисов, которые при сборке автоматически помещаются в системную директорию **~/.config/systemd/user**. Для управления реализованными демонами можно воспользоваться скриптами **start_daemons.sh** (запуск демонов) и **stop_daemons.sh** (остановка демонов) из корня проекта. 

Также запустить демонов можно вручную, использовав утилиту **systemctl**:

```bash
systemctl --user start <имя_узла>   # Запуск
systemctl --user stop <имя_узла>    # Остановка
systemctl --user status <имя_узла>  # Проверка состояния
systemctl --user restart <имя_узла> # Перезапуск
```

Имена узлов:
- **accel_server** - сервер
- **accel_node_a** - узел A
- **accel_node_b** - узел B

Для изменения номера порта сервера systemd-демонов необходимо пересобрать проект, изменив значение переменной **SERVER_PORT** в корневом **CMakeLists.txt**:

```CMake
set(SERVER_PORT "50051")
```

Для того, чтобы **systemd** считал изменения из нового файла, необходимо выполнить команду ниже:
```bash
systemctl --user daemon-reload
```

Теперь после перезапуска демона он начнет использовать другой порт.

Логи запущенных демонов можно посмотреть с помощью утилиты **journalctl**. Ниже приведен пример для просмотра логов сервера в режиме реального времени:

```bash
journalctl --user -u accel_server -f
```

# Обоснование выбора формата сериализации

В рамках реализации проекта был выбран формат **Protocol Buffers (Protobuf)**. Ниже представлен сравнительный анализ Protobuf с альтернативными форматами обмена данными в контексте задачи передачи потоковых данных акселерометра.

## Сравнительная таблица форматов

| Формат | Тип данных | Наличие IDL* | Скорость | Объем данных |
| :--- | :--- | :--- | :--- | :--- |
| **Protobuf** | Бинарный | Да | Высокая | Минимальный |
| **JSON** | Текстовый | Нет | Низкая | Высокий |
| **MessagePack** | Бинарный | Нет | Высокая | Средний |
| **CBOR** | Бинарный | Нет | Высокая | Средний |
| **Cap'n Proto**| Бинарный | Да | Максимальная| Высокий |

*\*IDL - язык описания интерфейса для генерации кода.*

## Описание форматов

**Protobuf** — это разработка Google, представляющая собой бинарный формат со строгой схемой данных. Он требует предварительного описания структуры сообщения, на основе которого генерируется эффективный код для упаковки и распаковки данных, обеспечивая высокую скорость.

**JSON** — текстовый формат, отличается человеко-читаемостью и гибкостью. Имеет низкую производительность и значительную избыточность, так как каждое сообщение содержит в себе имена всех полей в виде строк.

**MessagePack** — это бинарный формат, который можно назвать "бинарным JSON". Он сохраняет гибкость JSON (отсутствие жесткой схемы), но упаковывает данные плотнее.

**CBOR (Concise Binary Object Representation)** — формат, ориентированный на использование в устройствах с ограниченными ресурсами (Internet of Things). Он обеспечивает хорошую скорость и компактность, но его реализация на C++ зачастую сложнее и менее распространена в высокоуровневых распределенных системах по сравнению с аналогами.

**Cap'n Proto** — формат, имеющий максимальную производительность за счет представления данных в памяти так же, как в сети. Это делает его самым быстрым, но приводит к увеличению размера сообщений и усложнению архитектуры приложения.

## Итоговое обоснование выбора

Выбор **Protobuf** обусловлен его компактностью и скоростью: бинарный формат передает данные с датчиков гораздо быстрее и экономнее, чем обычный текст. Использование Protobuf вместе с gRPC позволяет автоматически генерировать код для сервера и клиентов, что исключает ошибки в структуре данных. Механизм номеров полей в сообщениях упрощает обновление протокола и поддержку разных версий узлов без поломки всей системы.**
